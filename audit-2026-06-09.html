Critical
CSS Architecture
C-02
design-system.css is never imported — dead code with conflicting values

src/styles/design-system.css (11.6 kB) defines --font-sans as system fonts, a complete spacing scale, and a different button shape (pill radius vs. radius-md in shared.jsx). It is documented in DESIGN_SYSTEM_UPDATE.md as "the new source of truth" — but no file in src/ imports it. The app loads: logbook.css (via main.jsx), injectGlobalCSS() (inline from shared.jsx), and sanctuary.css (via student-dashboard). If someone adds an import later, conflicting variable definitions will silently break styling. 

// main.jsx — what IS loaded
import './styles/logbook.css';        // ✓ loaded
injectGlobalCSS();                          // ✓ loaded (inline style tag)

// what is NOT loaded
// import './styles/design-system.css';   ← never imported
Decision needed: Either (a) delete design-system.css and DESIGN_SYSTEM_UPDATE.md if the inline CSS in shared.jsx is the authoritative source, or (b) migrate all CSS variables into design-system.css, add the import to main.jsx, and remove duplicates from shared.jsx's injectGlobalCSS(). Option (b) is better long-term.

High
CSS Conflict
H-03
Duplicate .dash / .dash-topbar rules in two CSS files produce unpredictable behaviour

Both logbook.css and sanctuary.css define .dash, .dash-topbar, .dash-brand-name, and related selectors with different values. logbook.css sets the dashboard background to var(--bg); sanctuary.css sets it to var(--sanctuary-parchment) (#f7f5f0). Both are loaded globally (logbook via main.jsx, sanctuary via the student-dashboard import which is not lazy). Whichever loads last wins — currently sanctuary wins because it is imported by a component that loads after the global CSS. If the import order changes, the student dashboard background silently changes. 

/* logbook.css */
.dash { background: var(--bg); }          /* #edf5f5-ish */

/* sanctuary.css — loaded after, wins */
.dash { background: var(--sanctuary-parchment); }  /* #f7f5f0 — warm parchment */
Fix: Consolidate. Move all student dashboard classes into sanctuary.css (the intended student-only stylesheet) and remove the duplicates from logbook.css. Or rename the student classes to use a .student- prefix throughout and avoid collision entirely.

High
Visual
H-05
Primary button gradient direction conflicts with the teal accent identity

The primary button style in shared.jsx:37 is: linear-gradient(135deg, var(--primary) 0%, var(--primary-ink) 100%). --primary is navy #1a2332 and --primary-ink is teal #247070. The result is a navy→teal gradient button with a teal glow shadow. On hover (line 38), it inverts to teal→navy. This reads as inconsistent: the platform's primary action colour appears to be both navy and teal depending on state. The brand colour system says teal is the accent, navy is the base — primary CTAs should be one solid identity, not a gradient between them.

Fix: Use a solid teal for the primary button: background: var(--accent) (#2d8b8b) with a subtle darken on hover. Reserve the navy for secondary "dark" CTAs where a heavier weight is needed. This aligns with the PRODUCT.md "calm, focused" brand — solid beats gradient for expert confidence.

Typography / Brand
M-02
met-platform-2.0.html uses Inter — explicitly listed as an anti-reference in PRODUCT.md

PRODUCT.md anti-references say: "Generic SaaS dashboards: cookie-cutter admin panels with default blue buttons." met-platform-2.0.html uses font-family: Inter, ui-sans-serif, system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif — Inter is the archetypal "generic SaaS dashboard" font. If this file is the design direction for version 2, it contradicts the brand. The live app uses DM Sans (correct and distinctive).

Fix: Either (a) delete met-platform-2.0.html as a stale exploration, or (b) if it represents the intended v2 redesign, update it to use DM Sans + DM Serif Display to stay aligned with the current brand font decisions.
Medium
Design Docs
M-03
DESIGN_SYSTEM_UPDATE.md documents an orange-accent system that does not match the live app

The update doc describes an orange accent (oklch(70% 0.15 50)) for waveforms, charts, and CTA elements. The live app uses teal everywhere. The doc lists button shapes as pill-radius, but the live app uses radius-md. This may be a future direction, a rejected exploration, or a doc written for a different version. Having an authoritative-sounding design doc that contradicts the running code confuses anyone trying to extend the system.

Fix: Add a note at the top of the file marking it as "proposed" or "archived", or delete it. If orange CTAs are the intended direction, update the live CSS to match.
Medium
Typography
M-04
font-weight: 800 used extensively — works but may undermine the "calm, expert" brand

Both logbook.css and shared.jsx use weight 800 for kickers, labels, and UI text in many places. DM Sans at 800 is quite heavy and wide. PRODUCT.md brand personality is "calm, focused, expert — quiet confidence over flashy energy." Heavy uppercase kickers at 800 weight can read as shouting, especially in a product aimed at adult learners. The teacher dashboard headline correctly uses DM Serif Display at 400 (italic-capable, editorial) — that is exactly right. The surrounding UI can afford to step down.

Fix: Audit all font-weight: 800 uses and reduce to 700 for most UI labels. Reserve 800 for the brand mark and the largest display headlines. Kickers (uppercase tracking labels) look better at 600–700 with wider letter-spacing.
Medium
Empty States
M-05
Empty state copy is polite but not actionable — from June 7 audit, still present

The June 7 audit identified that empty states say "No diagnoses yet" or "No classes scheduled today" but do not direct the teacher to the next specific step. From the teacher-dashboard source, empty todayClasses renders: "No classes scheduled today." — no link, no call to action, no "Add a class →" pointer. Same pattern repeats across Submissions and Diagnostics empty states.

Fix: Each empty state should include a ghost button or inline link pointing to the relevant creation flow. "No classes today. Schedule one →" for the calendar, "No diagnoses yet. Start after your next class →" for diagnostics.
Medium
Accessibility
M-06
Interactive div cards with onClick have no ARIA role or keyboard handler

Several cards in the teacher dashboard and KPI grid use div or a custom component with an onClick prop (e.g. the KPI cards at teacher-dashboard.jsx:81–83 with onClick: () => onNavigate('diagnostics')). These are not button elements, so they are not keyboard-reachable by default and are not announced as interactive by screen readers. The June 7 audit also flagged this pattern.

Fix: For any div/Card that accepts an onClick, either (a) render a <button> instead and style it appropriately, or (b) add role="button", tabIndex={0}, and a matching onKeyDown handler that triggers the action on Enter/Space. The Kpi and KpiCard components in shared.jsx should encapsulate this pattern once.

Polish and future-proofing

Low
Accessibility
L-01
Animations have no prefers-reduced-motion guard

shared.jsx defines @keyframes fadeUp, pulseDot, and edgeFlow. The teacher dashboard and KPI cards use animation: fadeUp 0.3s var(--ease) both with staggered delays. No @media (prefers-reduced-motion: reduce) block disables or reduces them. The PRODUCT.md accessibility target is WCAG AAA, which recommends respecting reduced-motion preferences. 

/* Add to the end of injectGlobalCSS() in shared.jsx */
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}
Fix: Add the above block. One addition, covers all animations in the app globally.
Low
Mobile

Confirm mobile login clip at 390 px is fixed (from June 7 audit)

The June 7 audit documented that the mobile login form clips horizontally at 390 px — the email field and sign-in button extending past the right edge. The June 2026 login code in login.jsx:67 defines .lp-signin-inner { width: min(100%, 420px); max-width: calc(100vw - 48px); } which should prevent overflow, but the previous screenshot captured the bug. This could not be verified without a live browser session.

Fix: Open http://localhost:4173 at 390 px viewport width and verify the sign-in form fits. If the clip persists, add min-width: 0; overflow-x: hidden; to .lp-root.

Strengths

What's working well

Good
Auth / Security
Auth is tight — teacher-only gate, PKCE flow, token validation on restore

App.jsx validates every restored session against Supabase, rejects unauthorized emails explicitly, handles both PKCE and implicit flows, and clears the URL immediately after consuming the code. The teacher/student role split is clean and the unauthorized path notifies the user with a friendly message. This is production-quality auth work.

Good
Student UX
Student feedback visibility rules are correct and responsible

student-dashboard.jsx only shows feedback marked sections.studentFeedback.approved === true. Unapproved diagnostic detail stays teacher-only. The MET concept glossary popups (MET_CONCEPT_GLOSSARY) turn opaque rubric terms into plain English inline — this is a genuinely useful and uncommon feature.

Good
Typography
DM Sans + DM Serif Display is a strong, on-brand font pairing

DM Sans (geometric humanist) for UI and DM Serif Display (elegant, editorial) for headlines is a coherent pairing that matches the "calm expert" brand. The teacher dashboard headline using DM Serif Display at regular weight is exactly the kind of quiet authority that PRODUCT.md describes. The --font-display-serif variable is the right call; use it more, not less.

Good
Navigation / A11y
Focus is moved to main content on SPA navigation — correct and rare

App.jsx:196–198 finds .shell-main, sets tabindex="-1", and calls focus() on every view change. This means screen reader users hear the new page announced without having to manually find it. Mobile nav items have aria-current="page". These are the right accessibility fundamentals — they just need the interactive-card ARIA fix (M-06) to complete the picture.
